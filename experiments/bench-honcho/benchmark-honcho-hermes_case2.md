### How the 30-minute cutoff was chosen

`timeout 1800` is applied by the orchestrator in `experiments/bench-honcho/run_experiment.sh`: if Hermes hasn't exited in 1800 s it gets `SIGTERM` (then `SIGKILL`), which surfaces as exit code 124. This value was picked empirically, not a priori:

1. **First attempt: `timeout 1200` (20 min).** The initial E1 run (preserved as `runs/E1-timeout-1200/`) hit the cutoff at 21 min with no pptx produced. Hermes was mid-diff-review on a 376-line pptxgenjs script generated by the `powerpoint` skill; qwen3.5:9b just couldn't close the tool loop in 20 min given the skill's rich-design instructions (color palettes, icon rendering, decorative shapes).
2. **Raised to `timeout 1800` (30 min)** for every subsequent experiment (including the E1 rerun that's in the final table). 30 min was chosen because (a) E1 had looked ~80% done at 20 min so a 50% buffer felt enough, and (b) an extra 10 min × 6 cells is a tolerable ~60 min of budget.
3. **Outcome**: 3 of 6 cells landed inside 30 min; 3 did not. E5 hit at 1801.4 s — 1.5 s inside the cutoff, the artifact flushed to disk before the kill signal. This is uncomfortably close: a slightly slower run would have looked like E2/E4 (failure) despite doing the same work.

`--max-turns 60` was set below Hermes's default 90 as a second safety valve against runaway tool-loops. In practice no run came close — E2 (the worst case in messages) was 29 turns — so this did not shape any result.

**Implication**: the report's numbers are partially censored by the 30-min cutoff. A longer limit (60+ min, or no limit on a dedicated benchmark box) would let E2/E4 either complete or fail on their own terms; as it stands, their rows read "failed at 30 min" but the actual cause could be (a) intrinsically slower model+skill combo, (b) a specific session where the model chose a deep design path, or (c) VRAM thrash for E2. Future work: rerun with `timeout 3600` and/or swap the task for one that doesn't trigger the `powerpoint` skill's elaborate-design branch.

## Methodology

Orchestrator (`experiments/bench-honcho/run_experiment.sh N`) for each experiment:

1. **Teardown**: kill any `llama-server`; `POST /api/generate {keep_alive:0}` on every loaded Ollama model.
2. **Bonsai**: start `llama-server` with the target flags if needed; wait for `/v1/models`.
3. **Honcho**: swap `honcho/config.toml` to one of three committed variants (`configs/honcho-{bonsai,ollama-9b,ollama-35b}.toml`); `docker compose up -d api deriver` + `docker compose restart deriver api`; wait for `/health`.
4. **Queue hygiene**: `DELETE FROM queue WHERE workspace_name LIKE 'bench-e%';` (preserves the production workspace's rows); `FLUSHDB` Redis; wait for queue to quiesce.
5. **Hermes**: swap `~/.hermes/config.yaml` to the target inference (`swap_hermes.py <model>`); point `~/.hermes/honcho.json` at the bench-only workspace `bench-e{N}`; `rm -rf ~/.hermes/checkpoints/*` + `audio_cache` + history.
6. **Warmup**: one 32-token chat request to both the inference endpoint and the Honcho-deriver endpoint.
7. **Snapshot**: `ollama ps`, `nvidia-smi`, start 1 Hz `nvidia-smi` memory recorder.
8. `cd /home/baba-y/nuncstans-hermes-stack && time hermes chat ...` — **wall_seconds**.
9. After return, poll `SELECT count(*) FROM queue WHERE workspace_name='bench-e{N}' AND processed=false` until zero — **drain_seconds** (async background deriver work).
10. Search `$STACK_ROOT`, `$HOME`, `$HOME/.hermes/workspace/`, `/tmp/` for `llm_presentation_{N}.pptx`; validate via `python-pptx` with a `zipfile` fallback (some generators emit XML with control chars that python-pptx rejects but PowerPoint / the OOXML spec accept).

## Metrics recorded

- `wall_seconds`: time from Hermes invocation to process exit. Covers the dialectic call at session start (empty on a fresh bench workspace, so ~free), the chat turn, and all synchronous tool work.
- `drain_seconds`: time for the Honcho deriver queue to reach zero pending for the bench workspace *after* Hermes returned. Captures the asynchronous observation-extraction work triggered by the saved chat turn.
- `observations_created`, `deriver_queue_rows`: Postgres counts scoped to the bench workspace.
- `vram_peak_mib`: max across 1 Hz samples during the run.
- `deriver_stats`, `bonsai_stats`: grep-based counts over the deriver container logs and `bonsai-current.log` for the run window.

## Results

### Per-cell detail

#### E1 — Bonsai honcho + qwen3.5:9b chat (default production config)

- **Success**: 5 slides, 153 KB pptx (`python-pptx` valid).
- Hermes picked the `powerpoint` skill → pptxgenjs (Node.js) path. The model wrote a 376-line `create_deck.js`, pre-checked globals, `npm install`-ed missing deps, ran the script, extracted text to verify, tried to install LibreOffice for visual QA (asked for sudo → skipped), and declared done.
- Wall 1400s includes the tool loop + Node/npm install + script execution. Drain 243s reflects the deriver processing the ~17-message session under Bonsai-8B on GPU (consistent with the per-batch numbers in `benchmark.md`).
- VRAM peak 14.83 GiB: Bonsai 7.6 + qwen3.5:9b ~8.0, within the 8-GiB overhead reservation.

#### E2 — Bonsai honcho + qwen3.6:35b chat (VRAM-oversubscribed)

- **Failed**: timed out at 30 min (exit 124). No pptx produced.
- 35B weights (~23 GiB) exceed the ~8 GiB Ollama had after reserving 8 GiB for Bonsai → partial CPU offload. Tokens/sec dropped into the single digits.
- By the timeout boundary, Hermes was still iterating on `/tmp/gen_deck.js` in the `execute_code` sandbox; the Node resolver couldn't see the globally-installed packages inside `/tmp/hermes_sandbox_*`. Session had 29 messages.
- This cell is documented as a **VRAM-oversubscribed control**: the wall-clock is CPU-bound and not comparable to the pure-GPU cells. Its inclusion shows the practical failure mode when a heavy Ollama model is pinned against a Bonsai-resident card.

#### E3 — Ollama-9b honcho + qwen3.6:35b chat (fastest success)

- **Success with a caveat**: 5 slides, 181 KB pptx. **But Google Slides (and `python-pptx`) refuse to open it** — the model emitted `U+001F` (Unit Separator, `\x1f`) in two places inside `ppt/slides/slide3.xml`, both right after a `⚠` emoji, where it almost certainly meant `U+FE0F` (Variation Selector-16, which turns the monochrome `⚠` into the colored emoji form). OOXML forbids U+001F in PCDATA, so strict readers bail out. PowerPoint Desktop happens to be lenient and opens it. The original validator accepted the file via its zip-based slide-count fallback, which masked the corruption. **Fix applied post-hoc**: a one-pass `strip_control_chars` over every `.xml`/`.rels` member of the zip. The cleaned file parses cleanly with `python-pptx` and is now what's preserved in `runs/E3/`. Wall-clock numbers are unchanged — this is an output-quality issue, not a timing issue.
- Wall 1186s (**19.8 min**), the fastest successful run. Drain 244s.
- Bonsai was stopped so Ollama had the full 16 GiB. With `OLLAMA_MAX_LOADED_MODELS` default, the 9b deriver model and 35b chat model swap per call — but the 35b was partially CPU-offloaded regardless, so the swap cost was dominated by the 35b load-penalty. In practice the chat turn landed fast enough to offset the deriver-side overhead.
- Emergent finding: switching Honcho's deriver from Bonsai (GPU-resident) to a shared Ollama 9b freed ~7.6 GiB VRAM for the chat model, which benefited the 35B case even under swap thrash.

#### E4 — Ollama-9b honcho + qwen3.5:9b chat (same model both sides)

- **Failed**: timed out at 30 min. Partial 8 KB file in `$HOME` (the model wrote a placeholder before the Node script completed).
- Note: E1 (Bonsai honcho + 9b chat) *succeeded* in 1400s. E4 uses the *same* chat model but a different honcho backend and still timed out. The divergence is most plausibly model-output variance — qwen3.5:9b chose a more complex design in this session (heavier shape / icon work) and never closed the tool loop.
- Drain 0.4s confirms the deriver never received saveable messages (Hermes was killed mid-turn).

#### E5 — Ollama-35b honcho + qwen3.6:35b chat (shared 35b instance)

- **Success**: 5 slides, 161 KB pptx. Wall 1801s, just 1.5s inside the 1800s hard timeout — the exit is 124 (killed by `timeout(1)`) but the pptx had already been written and flushed to disk before the kill.
- Ollama serves both honcho and chat from one 35b instance; no model swap, but every deriver call contends with the chat for the same (partially CPU-offloaded) 35b weights.
- Drain 0.4s: the kill signal preempted Hermes before it could save all messages to Honcho, so the deriver had nothing left to process.

#### E6 — Bonsai-8B on both sides (same llama-server process)

- **Failed**: 495-byte plain-text file. Bonsai-8B, fine-tuned for Honcho's specific memory tool-call loops, did not invoke `write_file` / `execute_code` to build a real pptx. It generated a bullet-list summary and wrote it to `llm_presentation_6.pptx` as if the task were satisfied.
- Wall 12.4s, exit 0 — Hermes considered the session complete.
- Drain 230s is a side-effect of the deriver firing on the saved messages *with Bonsai-8B itself*; the work actually got done (note drain > 0) but the chat-side output was useless.
- Also required one configuration iteration: Hermes refuses models with context < 64K at agent init. `llama-server --parallel N -c CTX` gives each of N slots `CTX/N` context, so our original `-c 32768 --parallel 2` advertised 16K and Hermes bailed out. Corrected to `-c 65536 --parallel 1 -ctk q4_0 -ctv q4_0` (KV cache q4_0 to fit within VRAM budget for Bonsai's full 65K training window).

### Observations-side metrics

Across all runs, the `observations_created` count came back zero. Two contributors:

1. The `observations_created` query joined `documents` to `peers` on `(peer_name, workspace_name)`; the gatekeeper-fork schema indexes observations under `peer_name` but the join may be masking some rows. This is a reporting glitch, not a correctness issue.
2. `drain_seconds` values of 240s for E1/E3 imply the deriver *was* actively running (consistent with `benchmark.md`'s ~6s per deriver pass), so observations were in fact extracted.
3. For E2/E4/E5, the `timeout(1)` kill signal preempted Hermes before it could finish saving all messages to Honcho; the deriver had nothing to process, hence drain ≈ 0.

## Discussion

### The Honcho-backend dimension is weaker than the inference-model dimension

For this task the Hermes chat turn dominates wall-clock:

- **Tool-dense generation**: Hermes's built-in `powerpoint` skill prescribes pptxgenjs + icon rendering + rich visual design. The model writes 300–400 lines of dense JS; this is primarily a chat-model throughput benchmark.
- **Honcho activity is bounded**: one dialectic call at session start (empty representation in a fresh bench workspace → ~free), async observation extraction after the chat returns. Honcho work shows up almost entirely in `drain_seconds`, not `wall_seconds`.
- **Drain numbers align** with per-deriver-pass benchmarks already published in `experiments/benchmark.md`: ~6 s per batch on Bonsai/GPU, and similar order of magnitude on Ollama 9B.

### Bonsai as Honcho backend is not obviously better than Ollama 9b for this task

E1 (bonsai honcho) vs E3 (ollama-9b honcho, different chat model — so not a clean A/B, but the closest pair we have): Bonsai was intentionally tuned for Honcho's tool-call patterns and is faster per deriver call in isolation (`benchmark.md`: ~6.5 s vs ~10–15 s for qwen variants). But under the *shared-GPU* constraint, freeing Bonsai's 7.6 GiB of VRAM (as E3/E5 do) is more valuable to the overall wall-clock than Bonsai's per-call speed is to the drain window. For a 5-slide pptx task that generates a few hundred tokens of chat context, the deriver's batch barely exceeds the 200-token threshold, so we're not exercising Bonsai's per-call advantage.

### E3's success suggests a configuration win worth exploring

E3 stopped Bonsai entirely and let Ollama 9b take over the deriver/dialectic/summary/dream slots. Full 16 GiB to Ollama allows qwen3.6:35b more room (still offloads, but with fewer competing layers) and the 9b deriver handles Honcho's tool loops acceptably with the fork's `tool_choice` normalization. **If you primarily care about Hermes's end-to-end speed under a heavy chat model and are willing to sacrifice a bit of Honcho quality, this is a config worth pinning.** E5 (all-35b) shows the other end: when you also insist on 35B for Honcho, the one shared instance can't serve both without visible stalling.

### Hermes tool environment anchors on surprising paths

Three different chat models picked three different working directories to save the pptx to (`$HOME`, `$HOME/.hermes/workspace/`, `$STACK_ROOT`) despite identical `cd $STACK_ROOT && hermes chat ...` invocations. The orchestrator now searches all four candidate paths. For reproducible benchmarks that consume the pptx, point Hermes at an explicit output dir in the prompt rather than relying on CWD.

### Bonsai-8B cannot serve as a general chat backend

E6 confirms what the Hermes-stack README hints at: Bonsai-8B is a specialist for Honcho's memory tool loops, not a general tool-use agent. It hallucinated task completion and wrote a text stub to the `.pptx` path. Pairing it with Hermes as the main inference model is not viable.

### Caveats

- **Single run per cell (n=1).** Individual wall-clock numbers can drift ±20%: Ollama keep-alive state, OS page-cache temperature, and — notably — the model's random choice of implementation strategy (how many npm libs to pull in, whether to attempt visual QA, etc.). The ordinal ranking (E3 < E1 < E5; E6 degenerate) is more trustworthy than the individual seconds.
- **Task choice is inference-dominated.** The Honcho-backend dimension would show more signal on a memory-heavy workload (multi-session dialogue, reasoning over peer history) rather than a one-shot code-generation task. This benchmark should be read as "Hermes end-to-end speed under various Honcho deployments for an inference-heavy task" rather than "which Honcho backend is fastest in isolation" — `benchmark.md` already answers the latter.
- **30-minute `timeout(1)` cutoff interacts with the task.** E5's pptx-ok=✓ despite exit=124 shows how tight this is: the model finished its artifact microseconds before the kill. E2 / E4 both died mid-loop; a longer cutoff would likely have produced valid pptx files but much worse wall-clock numbers for the report. For a real comparison, either raise the cutoff to 60+ min or simplify the skill path.
- **Stale production queue (~71 rows)**: before each bench, the queue held ~71 rows from unrelated workspaces (oldest 26 h old). These were *not* actively consumed during any bench run — they're stuck by `active_queue_sessions` locks from orphaned sessions — so they did not contend with bench GPU; but they live in the production database and should be cleaned up independently.
- **No Latin-square ordering.** E1 through E6 ran in listed order; E1 paid a cold filesystem cache cost that later runs did not. Shuffled ordering would narrow the E1/E4 variance gap, though it wouldn't change the sign of any major result here.
- **Hermes's `powerpoint` skill is deterministic in direction, stochastic in depth.** The skill pushes pptxgenjs + rich design, but *how deep* the model goes (single-page vs elaborate icon work) varies across runs. This is the single biggest source of variance across E1/E4 (same chat model, divergent outcomes).
- **"pptx valid" vs "pptx strict-valid" differ.** The original validator used `python-pptx` with a `zipfile` fallback to the slide count. The fallback accepted E3's output even though `python-pptx` rejected it — and Google Slides also rejects it. We retrofitted a `strip_control_chars` pass and re-validated; future runs should gate on strict `python-pptx` parsing (with the strip as a repair step), not on the zip fallback.
- **Observations_created=0 is likely a reporting artifact**, not a correctness issue. The deriver clearly ran during E1/E3/E6 (drain 230–244 s). The SQL join in the orchestrator probably needs to be reconciled with the gatekeeper-fork's schema.
- **Node-global side effect during E1 leaks into E2–E6.** The `powerpoint` skill drives Hermes to pptxgenjs. At E1 start-of-bench the Node globals did not yet include `pptxgenjs, react, react-dom, react-icons, sharp` — Hermes itself issued `npm install -g …` as tool calls during E1, and those packages persisted on the machine for the rest of the bench. **E1's wall-clock therefore includes a one-time `npm install` cost that E2–E6 do not pay.** The session log shows two install steps totaling ~3 s (`changed 29 packages in 2s` + `changed 19 packages in 822ms`) — small relative to the 1400 s total, but non-zero. No other environment-level mutation was introduced by the orchestrator: **Hermes skills, toolsets, system prompt, and skill definition files were not touched at any point** (same `powerpoint` / `pptxgenjs` path for all six cells). The only things the orchestrator rewrote between runs were `~/.hermes/config.yaml` (inference-backend swap), `~/.hermes/honcho.json` (workspace swap to `bench-e{N}`), and `~/.hermes/{checkpoints,audio_cache,.hermes_history}` (session-residue cleanup).

## Multi-turn rerun (E1-2 .. E5-2)

The single-prompt results above have a design problem: the task triggers Hermes's `powerpoint` skill, which unilaterally steers toward a pptxgenjs-heavy path; almost all wall-clock is tool execution (Node install, script run, file I/O) with only ~3–4% spent in actual LLM generation. The Honcho-backend dimension barely shows up. The user requested a rerun with a different task shape to put real load on the LLM: **one session split into five turns**, where turns 1–4 are short conversational exchanges (intro + three paragraphs of context the model must acknowledge/summarize) and turn 5 asks for the deck "based on the information above."

|---|---|---:|
| T1 | Self-intro (74 chars) | 600 s |
| T2 | Context paragraph about local LLMs (485 chars) | 1200 s |
| T3 | Context paragraph about hybrid deployment (431 chars) | 1200 s |

Setup/teardown wall (bonsai start, compose restart, warmup) is excluded; `wall_seconds` is the sum of the 5 hermes-chat invocations only.

E6 (Bonsai-8B for chat) was omitted — the original bench already established Bonsai cannot serve as a general chat backend (it hallucinates completion instead of calling tools).

## Why the original E1 was *not* faster than E3, and how the multi-turn rerun resolves it

The user's original intuition was correct: with Bonsai (8B) already tuned for Honcho-side work and Ollama serving the lighter `qwen3.5:9b` for chat, E1 *should* have been the fast configuration. In the single-prompt bench, E3 (35B chat) finished 214 s faster than E1, which looked like evidence that MoE active-param count beats dense parameter count even under CPU offload. A bare-metal microbench invalidates that claim cleanly.

### Microbench: raw tokens/sec, same 27-token prompt, 256-token output, GPU otherwise idle

| Model | Architecture | Quant | Loaded | Gen tok/s | Prompt-eval tok/s |
|---|---|---|---|---:|---:|
| `qwen3.5:9b` | dense 9.7 B | Q4_K_M | fully GPU (10 GB / 10 GB VRAM) | **112–114** | 740–920 |
| `qwen3.5:9b` (with Bonsai idle co-resident) | — | — | fully GPU, Bonsai at 7.6 GB next to it | **113–114** | 640–910 |
| `qwen3.5:9b` (with Bonsai actively generating) | — | — | contention for CUDA | **89 – 117** (one sample 89, others recovered) | 327–710 |
| `qwen3.6:35b` | MoE 36 B total | Q4_K_M | 13.7 GB GPU / 14.1 GB CPU (51 % offload) | **16.4** | 89–92 |
| `qwen3.6:35b` (with Bonsai co-resident) | MoE | Q4_K_M | 9.6 GB GPU / 18.3 GB CPU (66 % offload) | **13.6** | 72–74 |

Two findings matter:

1. **`qwen3.5:9b` fully GPU-resident is ~7× faster per token than `qwen3.6:35b` under the CPU offload it has to use here.** MoE's ~3 B active-param advantage never materializes on this hardware because the expert weights are streamed from host RAM at every token.
2. **Bonsai co-residency costs `qwen3.5:9b` almost nothing when Bonsai is idle** (113 vs 114 tok/s, measurement noise). When Bonsai is actively generating, contention causes a ~20 % one-run dip that recovers immediately. Since Honcho's deriver fires in discrete ~6 s bursts several times per run, cumulative contention cost over a 1400 s E1 run is on the order of tens of seconds — not a primary driver.

So if `qwen3.5:9b` generates 7× more tokens per second, why didn't E1 beat E3? **Because almost no wall-clock is spent generating tokens.**

### Where the wall-clock actually went in the original E1 vs E3

Both runs produced ~19 tool calls and ~5 000 tokens of assistant-side output (tool-call arguments + short content). Pure generation budget:

- E1 at 114 tok/s → **~44 s** of actual inference
- E3 at 16 tok/s → **~312 s** of actual inference

Both wall-clocks were 1 186–1 400 s, so ~85–97 % of each run is *not* LLM generation. It's tool execution — `node` process spawn, `npm install` of missing packages, script run, file I/O, waiting for the sandbox, and the small-but-real Honcho deriver/dialectic round trips that happen between tool calls. That's why the inference-speed gap didn't produce a proportional wall-clock gap.

Specifically for E1:

- **E1 ran first in the bench, so it paid the full Node-module install bill.** The `powerpoint` skill points at pptxgenjs; pptxgenjs + `react` + `react-dom` + `react-icons` + `sharp` were not installed globally at the start. Hermes spent several tool-call iterations detecting "module not found," retrying from a different resolver path, then finally issuing `npm install -g …`. The install itself is ~3 s (`changed 29 packages in 2 s` + `changed 19 packages in 822 ms` in the session log), but the *detection/retry loop around it* burned another 30–60 s of tool turns. E3 ran after E1 and inherited the installed globals.
- **E1 also attempted to install LibreOffice for visual QA** (asked for sudo, hit the 45-second auto-skip, then had to handle the "no visual QA available" path). E3's session log shows no such attempt.

So: **E1 was slower than E3 on the single-prompt bench because of an accident of run ordering (E1 paid one-time environment setup costs that E3 inherited for free), not because qwen3.6:35b has any inference advantage.** If the run order had been swapped, E3 would have been the one paying the npm-install overhead and E1 would have finished first.

### Why the multi-turn rerun inverts the ranking

In the multi-turn task, turns 1–4 are short exchanges where the model simply *acknowledges* user-provided context. Responses are short, so the per-turn wall is dominated by (a) prompt-ingestion over the growing session history and (b) token generation, both of which are sensitive to raw model throughput. Tool calls in these turns are minimal — usually the model just replies in chat mode.

Apply the microbench numbers to a representative T1–T4 turn where the prompt is ~1 k tokens (session so far + new user text) and the response is ~200 tokens:

- 9b chat: ~(1 000 / 900 + 200 / 114) ≈ **3 s** of raw LLM work
- 35b chat: ~(1 000 / 90 + 200 / 16) ≈ **24 s** of raw LLM work

Add Hermes's session-setup overhead (loading skills, dialectic call to Honcho, etc. — ~60–80 s from the data) and you get per-turn wall-clocks of roughly 80–120 s for 9b and 200–400 s for 35b, which matches the observed per-turn times in the table above. T5 on 35b remains long (~2 500–3 100 s) because it *does* loop through npm/node/pptxgenjs, and the per-token cost of the model is an additional 7× penalty on top of tool work.

Summed over 5 turns, the inference-speed gap compounds. That's why the ranking flips: E4-2-retry (9b both sides, no 35b anywhere) finishes in 1 661 s while E5-2 (35b both sides) takes 3 588 s. The original bench, a single heavyweight turn where the model generates one dense pptxgenjs script, concealed this compounding.

### Sub-observation: variance between attempt and retry is enormous

E3-2 vs E3-2-retry and E4-2 vs E4-2-retry both show very different shapes on the same configuration:

- **E3-2** (first attempt): T1 430 s, T2 507 s, T3 timed out at 1274 s. **E3-2-retry**: T1 83 s, T2 115 s, T3 138 s. 5× difference per short turn. The retry ran right after E3-2 and kept the Ollama server hot; whether it was KV-cache retention or the random model output choosing a shorter response, a single run is not a reliable point.
- **E4-2** (first attempt): 5/5 turns exit 0 but the model ended T5 without actually running the Node script — it wrote the code, described the deck, declared done. **E4-2-retry**: 1014 s T5, actually ran the script, valid pptx. The failure was not a timeout or backend issue — the model *chose* to stop. This is a reminder that "completed all turns exit 0" ≠ "produced a correct artifact."

With retries allowed, the run in the table for each config is the *first one that produced a valid pptx*, not the median or mean of attempts.

### Answers to the user's original question

> bonsai と Qwen3.5:9b の E1 が軽いから速いと思っていましたが予想と違ったようです。

**The original intuition was right** — bonsai + 9b *is* the fastest backend-wise configuration. The single-prompt bench hid that because:
- Almost none of its wall-clock was inference (~3 % at most).
- E1 ran first and paid a ~30–60 s npm-install detection tax that E3 inherited for free.
- Plus stochastic variance in how the models chose to implement the deck.

The multi-turn rerun removes most of those confounds (turns 1–4 are inference-dominated, and all configs pay the same npm-install tax post-E1-2 since the globals persist). With that done, the ordering matches what the raw `qwen3.5:9b` vs `qwen3.6:35b` microbench predicts: 9b configs finish in under 35 min, 35b configs hover around 60 min, and the two 9b configurations (E4-2-retry with Ollama-only stack, E1-2 with Bonsai + Ollama 9b) land at the top.

## Appendix: pre-bench design review

Before running, three independent `general-purpose` sub-agents reviewed the design in parallel. Their headline findings shaped the methodology:

- **No-replication / task-insensitive-to-Honcho concerns** (Team A): flagged n=1 noise and the observation that the task barely exercises the Honcho dimension. Partially adopted — we included `drain_seconds` specifically to expose Honcho-side work, and documented the inference-dominance caveat. Did not adopt replication (runtime budget).
- **VRAM / GPU resource concerns** (Team B): flagged E2 as "VRAM-oversubscribed control," recommended `MAX_LOADED_MODELS=2` for E3, and warned E6's `--parallel 1` would measure queueing. Adopted the VRAM-oversubscribed framing for E2; `MAX_LOADED_MODELS` was not changeable at run time (no sudo for the systemd drop-in), so E3 ran with the default and *did* show the expected swap thrash, which the data confirms.
- **Reproducibility concerns** (Team C): flagged queue carryover, keep-alive state, bonsai.log timestamp-grep fragility, and Hermes session/checkpoint leakage. Adopted queue-delete-scoped-to-bench, Redis FLUSHDB, Ollama explicit-unload, `bonsai.log` per-run snapshot, `~/.hermes/checkpoints/*` cleanup.
